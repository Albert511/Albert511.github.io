# 1、GitHub中的gmeek不自动构建了
刚开始编辑一条issue并保存可以自动构建，然后访问网址可以访问到保存的信息。时间久了之后会出现新增或者编辑issue不能自动构建的现象。
## 原因：
GitHub 的规则是：
如果仓库 60 天没有 push / commit / issue / PR 等活动，所有使用 schedule 触发的 Actions 会被自动 disabled。
## 解决方案
### 提交一次 commit
只要仓库有 activity 就会恢复。
比如：
`git commit --allow-empty -m "reactivate workflow" 
git push`
或者随便改个 README：
`git add .
git commit -m "update"
git push`
然后 workflow 会恢复。
### 在GitHub网页直接修改一个文件（推荐）
> 推荐修改README.md然后直接提交。

步骤：
1、打开你的仓库主页
2、找一个文件（比如 README.md）
3、点击右上角 ✏️ Edit
4、随便加一行，比如
`update workflow`
5、点击 Commit changes
这样 GitHub 就会产生一次 commit activity，workflow 就会恢复。

# 2、服务器中Redis操作失败

测试环境的服务调用接口报错，根据返回的错误码知道是操作Redis时报错，根据返回的traceId定位日志，发现错误日志是跟aof（Redis的备份机制）有关的，服务器磁盘空间不足，会导致aof文件的写入有问题。立马查看服务器磁盘的占用情况，发现是docker构建镜像时的缓存文件占用了大量空间，然后清理掉该部分的缓存空间后，Redis恢复。

# 3、服务调用第三方服务报错

三方服务的接口访问是https的，在本服务的接口中调用也是使用https的方式调用三方接口，显示ssl有关的握手异常，但是之前访问其它三方服务的https接口也是没有问题，然后排查了https支持的哪些版本，发现本服务中最高是到TLS1.2的，推断第三方服务的接口可能高于这个，然后尝试验证高版本的TLS，在本地测试环境中是可以的，发布到测试环境时报错了，还是访问不通，接着排查发现本地开发的jdk8是421的，但测试服务器中docker中的jdk8是2xx，再查资料发现4xx的某个版本后就支持了TLS1.3，但是2xx的版本还是不支持，然后再找高于4xx的镜像，各种找，最后还是找到了，以为都没有问题了，但是发布到测试环境还是有问题，又发现该版本的jdk缺少了某些加密算法，然后再添加jdk8有关的加密的依赖，最后终于敲定这个问题。

# 4、IDEA 单元测试报错

## 1、运行单元测试时提示下面错误

<img width="417" height="131" alt="Image" src="https://github.com/user-attachments/assets/ad4c6bbe-b0a9-41f0-8885-20173fb80bc5" />

```
Error running
DeviceServiceOuterTest.testGetWeatherlnfoForDeviceByDeviceld.
Command line is too long.
Shorten the command line via JAR manifest or via a classpathfile and rerun.
```

### 背景：

之前的IDEA版本是2022.3.3版本，单元测试是没有问题的，升级到2025.3.4后，运行单元测试就会报这个错误。

### 问题排查：

这个现象非常符合：

> IDEA 从 2022 升级到 2025 后，JUnit/Test 的 classpath 处理机制变化导致的。

### 问题处理：

打开运行配置，修改配置模版，依次找到并修改下面三项

* JUnit 
* Application
* Spring Boot

找到Shorten command line设置为classpath file，然后点击apply，点击OK；
